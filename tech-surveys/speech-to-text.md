# 한영 코드스위칭 STT 개발 전략

> 온프레미스 상용화를 위한 음성인식(STT) 시스템 개발 방향 문서
>

---

## 1. 프로젝트 개요

| 항목 | 내용 |
| --- | --- |
| 목표 | 한영 코드스위칭 대응 상용 STT 시스템 구축 |
| 대상 언어 | 한국어 / 영어 (한영 혼용 발화 중심) |
| 배포 환경 | **On-premise** (자체 GPU 서버) |
| 목표 WER | 3% (도메인 한정 기준) |
| 베이스 모델 | NeMo Fast-Conformer / Canary |

---

## 2. 기술 선정 배경

### 2.1 Whisper를 채택하지 않은 이유

- Encoder-Decoder 구조로 **실시간 스트리밍 불가**
- 대형 모델 기준 높은 추론 비용 (상용 트래픽 대응 부담)
- 청크 처리 시 문맥 경계 문제 발생

### 2.2 채택 아키텍처

- **CTC Chunked** 및 **Conformer-Transducer(RNN-T)** 계열
- 청크 단위 처리 + 진정한 온라인 추론 지원

---

## 3. 모델 비교 및 선정

### 3.1 Fast-Conformer vs Canary

| 기준 | Fast-Conformer (RNN-T) | Canary (AED) |
| --- | --- | --- |
| 라이선스 | Apache 2.0 | CC-BY (버전별 상이) |
| 실시간 스트리밍 | 강점 | 약함 (offline) |
| Offline 정확도 | 높음 | 더 높음 |
| 코드스위칭 기본기 | 데이터 의존 | 멀티태스크 강점 |
| 번역 동시 지원 | 미지원 | 지원 |
| 파인튜닝 용이성 | 쉬움 | 어려움 |
| 온프레미스 최적화 | TensorRT 성숙 | 가능 |

> **참고:** Canary는 내부적으로 Fast-Conformer Encoder를 사용함. 본질적 차이는 디코더 구조(RNN-T vs Attention Decoder).
>

### 3.2 선정 전략 (하이브리드)

```
1. Canary로 빠른 PoC → 한영 성능 베이스라인 측정
        ↓
2. 병행하여 Fast-Conformer 자체 파인튜닝 진행
        ↓
3. 최종 상용은 Fast-Conformer로 통일
   (Apache 2.0 + 스트리밍 + 최적화 성숙)
```

**결론**

- 빠른 검증 → **Canary** (baseline 측정 전용)
- 상용 배포 → **Fast-Conformer** (라이선스 자유, 스트리밍, 경량화)

## 4. 코드스위칭 대응 설계

언어별 분기 모델은 코드스위칭에 **부적합** (문장 내 언어 혼용 시 Language ID 분기 붕괴).

### 핵심: 통합 모델 + 통합 Vocabulary

```
오디오 스트림
    ↓
Silero VAD
    ↓
Fast-Conformer Encoder (단일, 언어 구분 없음)
    ↓
RNN-T Joint Network
    ↓
통합 BPE Vocabulary (한글 자모 + 영문 + 공통)
    ↓
출력: "이 model을 deploy해주세요"
```

- 단일 SentencePiece 토크나이저 (vocab ~8000)
- `character_coverage ≈ 0.9995`로 한글 자모 완전 커버
- 한영 혼용 코퍼스로 직접 학습 → 자연스러운 토큰 경계

## 5. WER 3% 달성 전략

> **현실 인식:** 범용 한영 자유발화에서 WER 3%는 SOTA로도 어려움.
**도메인을 좁히고 + 자체 데이터를 충분히 확보**해야 3~5% 구간 도달 가능.
>

### 5.1 데이터 확보 (가장 중요 / 병목)

| 데이터 | 규모(추정) | 비고 |
| --- | --- | --- |
| 일반 한국어 | 1000h+ | AI Hub 등 |
| 일반 영어 | 1000h+ | LibriSpeech 등 |
| 한영 코드스위칭 | 100~500h | **자체 수집 필수** |
| 도메인 특화 발화 | 50~200h | **결정적 요소** |

> WER 3% 달성의 80%는 데이터 품질/양이 좌우함.
>

### 5.2 정확도 개선 수단

- **후처리 LM**: n-gram Shallow Fusion (WER 2~4%p 개선)
- **도메인 사전 교정**: 영어 전문용어 정규화
- **표기 정규화**: `AI`↔`에이아이`, `GPU`↔`지피유` 통일 규칙

## 6. 온프레미스 배포 스택

```
하드웨어
└── GPU 서버 (A10G / L40S / A100급)

추론 엔진
└── NVIDIA Triton + TensorRT
    ├── Fast-Conformer → TensorRT 변환
    └── 동시 스트림 배칭 처리

서빙
└── gRPC / WebSocket 스트리밍 엔드포인트

모니터링
└── WER 샘플링 검증 + 내부 로깅
```

**성능 목표**

- RTF(Real Time Factor) < 0.3
- 청크 레이턴시 < 300ms
- A10G 1장 기준 동시 ~50 스트림

## 7. 단계별 로드맵

| Phase | 기간 | 내용 | 예상 결과 |
| --- | --- | --- | --- |
| **Phase 0** | 2~4개월 | 코드스위칭·도메인 데이터 수집/라벨링 ★병목 | 학습 데이터 확보 |
| **Phase 1** | 1~2개월 | Fast-Conformer + 통합 BPE 학습 | WER 10~15% |
| **Phase 2** | 2~3개월 | 도메인 파인튜닝 + LM 결합 + 정규화 | WER 5~8% |
| **Phase 3** | 1~2개월 | TensorRT + Triton 온프렘 배포 | 도메인 한정 WER 3~5% |

---

## 8. 핵심 결론

1. **상용 베이스 모델은 Fast-Conformer** (Apache 2.0, 스트리밍, 경량화)
2. **Canary는 PoC/베이스라인 측정용**으로 병행 활용
3. **코드스위칭은 통합 모델 + 통합 BPE**로 대응 (언어 분기 X)
4. **WER 3%의 관건은 데이터** — 도메인 한정 + 자체 데이터 확보가 필수
5. 후처리 LM·정규화로 추가 WER 개선
